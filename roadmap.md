# D-OPSD Trainer Roadmap

## Goal

Build a local D-OPSD model trainer with a clear, modern Liquid Glass UI, stable job execution, dataset validation, model management, and repeatable environment setup. The trainer uses `vvvvvjdy/D-OPSD` as the training core and borrows workflow ideas from `ostris/ai-toolkit`: dashboard, datasets, new job wizard, queue, logs, GPU monitor, samples, and settings.

## Current Reality

- Workspace root is the D-OPSD trainer workspace.
- The upstream D-OPSD project is script-first and has no native UI or job manager.
- Training scripts are Linux/Accelerate/DeepSpeed oriented; WSL2 Ubuntu is the preferred training backend on this Windows host.

## Product Direction

The first screen is the trainer dashboard, not a marketing page.

Primary navigation:

- Dashboard: GPU status, model cache status, active queue, recent jobs, disk space, environment health.
- New Job: recipe-first D-OPSD training wizard with memory and launch controls.
- Jobs: queue, run history, controls, logs, loss, samples, checkpoints.
- Datasets: JSONL upload/import, image path validation, prompt preview, bucket summary.
- Models: required model downloads, license/auth status, cache size, integrity checks.
- Settings: WSL/Python/Conda paths, Hugging Face token status, output folders, runner policy, model readiness, low-VRAM safety defaults, and verified 16GB starter profiles.

## UI Design Direction

Use a restrained Liquid Glass production-tool style:

- Glass for side navigation, top command bar, status chips, modals, and primary action surfaces.
- Solid high-contrast surfaces for tables, forms, logs, charts, and image grids.
- Avoid stacked glass panels and decorative-only blur.
- Keep rounded corners at 8px or less for dense tool surfaces unless a modal needs slightly more softness.
- Use lucide icons for buttons and navigation.
- Every main screen must be understandable without instructional filler text.
- Buttons, inputs, sliders, toggles, menus, tabs, tooltips, loading states, empty states, and error states must be functional.
- Memory-sensitive recipe screens must expose launcher choice, low-VRAM offload, 8-bit Adam, sample/checkpoint skipping for smoke runs, resolution scale, and a command preview.
